# EditBench: Evaluating LLM Abilities to Perform Real-World Instructed Code Edits

> **Venue**: ICLR 2026 | **ArXiv**: 2511.04486 | **EditBench**

近年来，大型语言模型（LLM）在软件工程领域的应用日益广泛，以 GitHub Copilot、Cursor 为代表的 AI 编程助手已成为开发者日常工作的重要工具。这类助手所承担的核心任务之一，便是"指令式代码编辑"（Instructed Code Editing）——即根据用户以自然语言给出的指令，对开发者现有代码进行定点修改。然而，尽管 HumanEval、CodeXGLUE 等代码生成基准已相当成熟，针对代码编辑这一更贴近真实使用场景的能力却几乎没有系统性评估框架。现有的少数相关数据集往往依赖合成或人工构造的示例，缺乏对真实开发者工作流程的忠实还原，从而导致基准与实际应用之间存在显著的分布鸿沟。正是为了填补这一空白，Wayne Chi 等来自卡内基梅隆大学和加州大学伯克利分校的研究者构建了 EDIT-Bench，一个完全基于真实开发者使用数据的指令式代码编辑基准，并在 ICLR 2026 以口头报告（Oral）的形式发表。

EDIT-Bench 的核心构建理念是"从真实世界中采集"。研究者开发了一款 VS Code 扩展插件，并将其部署给近 500 名真实开发者使用。每当开发者在编辑器中借助 AI 助手执行代码修改操作时，插件便会记录下完整的交互信息，包括用户输入的自然语言指令、当前光标所在位置、用户高亮选中的代码片段、以及完整的文件上下文。经过清洗与筛选，最终形成了包含 540 个问题的基准数据集（论文中某处引用为 545 个）。这些问题跨越 5 种自然语言（英语、中文、日语、西班牙语、法语等）以及 2 种主要编程语言，涵盖从修复错误、重构代码到添加新功能等多种真实使用场景。为了兼顾评估效率与全面性，研究者同时提供了一个由 108 个问题构成的"核心子集"，供快速验证使用，以及完整的 540 题集合，用于系统性综合评估。

EDIT-Bench 在方法论层面最重要的创新之一，是将"上下文依赖性"（context-dependence）明确纳入评估设计。不同于代码生成任务中模型仅需依据问题描述从零开始输出代码，指令式代码编辑本质上是一种条件修改任务，其输出必须与既有代码库保持一致性、风格连贯性和逻辑协调性。具体而言，EDIT-Bench 中的每个问题都包含四个信息维度：用户的自然语言指令 $I$、当前文件的完整代码上下文 $C$、用户选中高亮的代码区域 $H$、以及光标位置 $P$。研究者通过系统性地控制这四个维度的完整程度，构造了不同"上下文配置"（context configuration）下的评估实验，从而量化每一类上下文信息对模型性能的具体贡献。实验结果表明，不同上下文配置之间的性能差异最高可达 11 个百分点，充分说明上下文信息的完整程度对于代码编辑任务的成功率具有决定性影响。这一发现对实际产品设计也具有重要启示：AI 编程助手在向后端模型传递请求时，应尽可能携带完整的上下文信息，而不应简单截断。

在评估指标的选取上，EDIT-Bench 以通过率（pass rate）为核心指标，即模型输出的修改结果是否与标准答案匹配或通过测试的比例。由于代码编辑任务的输出空间远比代码生成更具结构约束——模型需要在保留未修改部分的同时对指定区域进行精准更改——单纯的精确匹配（exact match）往往过于严苛，而功能正确性（functional correctness）又依赖于可运行的测试用例，因此研究者综合运用多种粒度的评估手段以获得更全面的视图。基准数据集与评估代码均已开源，托管于 HuggingFace 和 GitHub（waynchi/editbench），以支持社区的复现与扩展。

实验部分对 40 款不同规模与来源的大语言模型进行了系统性评估，结果揭示了当前模型在指令式代码编辑上的整体局限。根据项目官网公布的排行榜数据，当前表现最优的模型为 Claude Sonnet 4，其完整数据集通过率约为 64.81%，其次是 Claude Sonnet 4.5（59.81%）和 Claude 3.5 Sonnet（59.07%）。GPT-4o 的通过率约为 53.33%，Llama 3.1 405B 约为 48.70%，Qwen 2.5 72B 约为 45.19%。整体来看，在 40 个被评估模型中，仅有极少数（约 1 至 3 个）超过了 60% 的通过率，大多数模型表现低于此阈值，表明 EDIT-Bench 是一个对当前模型具有真实挑战性的基准。模型在不同指令类别下的表现也存在显著差异：错误修复类任务通常比功能添加类任务更难，因为前者要求模型精准定位并修复语义错误，而非简单追加代码。此外，跨越多个代码块的复杂编辑请求普遍导致模型性能下降，常见失败模式包括引入不必要的修改、破坏缩进格式、以及无法正确维护嵌套结构中的代码语义。

从更宏观的视角审视，EDIT-Bench 的意义超越了一个新基准本身。它所确立的评估范式——从真实开发者行为中采集数据、在受控上下文变体中系统评测、以通过率而非启发式相似度为核心指标——为未来代码编辑领域的基准研究提供了重要参照。更关键的是，该工作揭示了现有 LLM 评估体系中一个长期被忽视的维度：模型在"修改已有代码"上的能力与在"生成全新代码"上的能力之间，可能存在相当大的差距，而这一差距在真实开发场景中直接影响用户体验和生产效率。随着 AI 编程助手从代码补全向深度代码重构演进，像 EDIT-Bench 这样面向真实编辑场景的评估工具将扮演越来越重要的角色，推动学术界和工业界在这一方向上持续优化模型能力。
