# Language Confusion Gate: Language-Aware Decoding Through Model Self-Distillation

> **Venue**: ICLR 2026 | **ArXiv**: 2510.17555 | **LanguageConfusionGate**

大型语言模型在多语言场景下的广泛部署催生了一个长期存在却难以彻底解决的问题——语言混淆（language confusion），即模型在生成文本的过程中无意间将非目标语言的词汇或字符混入输出。这一问题在实际应用中普遍存在：即便是当前最先进的模型，如 GPT-5-Chat，仍会在中文或日文任务中产生约 0.57% 的中日字符混淆，以及约 0.67% 的拉丁字母混淆。尽管比例看似不高，但在长文本生成、机器翻译、代码生成等高要求场景中，这类错误严重影响输出可读性和用户体验。更复杂的是，语言混淆与合理的代码切换（code-switching）难以区分——例如，在翻译非拉丁语言时引用英文专有名词或编程关键字是完全合理的行为。现有解决方案要么需要对模型进行全量重新训练（成本高昂），要么依赖提示工程（效果有限），要么采用强化学习微调方法（如 ORPO），但这类方法往往以牺牲模型通用能力为代价。来自阿里巴巴研究团队的 Collin Zhang、Fei Huang、Chenhan Yuan 和 Junyang Lin 提出了语言混淆门（Language Confusion Gate，LCG），一种轻量级、即插即用的解码干预方案，在不修改底层模型权重的前提下，通过对解码阶段进行选择性 token 过滤来消除语言混淆。

LCG 的核心设计建立在三个关键实证观察之上。第一，语言混淆在正常生成中是偶发性的，而非持续性的，说明模型本身在大多数时刻能够正确识别目标语言，仅在少数位置发生偏差。第二，在这些发生混淆的位置，正确语言的 token 仍有 99.29% 的概率出现在模型预测的前三候选中，这意味着只需从候选集中过滤掉错误语言的 token 即可修复大多数错误，而无需重新生成整个序列。第三，高资源语言的 token 嵌入向量具有更大的二范数，从而在解码时获得系统性的概率优势，导致低资源语言任务中更容易引入高资源语言的 token。这一观察基于对 logit 计算公式的分解：对于 token $i$，其 logit 为 $\text{logit}_i = \|h\| \cdot \|e_i\| \cdot \cos(h, e_i)$，其中 $h$ 为最终隐藏状态，$e_i$ 为 token 嵌入向量。由于高资源语言（如中文、英文）的 $\|e_i\|$ 系统性地大于低资源语言的对应值，即便余弦相似度相当，高资源语言的 token 也会在 softmax 竞争中占据优势。以 Qwen3-8B 为例，10.74% 的中日 token 的嵌入范数位于全词表前 5%，而低资源语言 token 这一比例仅为 0.14%，差距极为显著。

基于上述分析，LCG 被设计为一个轻量级的双层 MLP，以 LLM 最后一层的隐藏状态 $h_t$ 作为输入，输出四个语言族的 logit：中日（CJ）、拉丁（Latin）、符号（Symbols）和低资源语言（Low-Res）。在推理阶段，LCG 预测当前位置应当属于哪些语言族，并对不属于预测语言族的 token 进行掩码屏蔽，从而阻止模型从这些 token 中采样。为了训练这个门控模块，论文提出了规范调整自蒸馏（norm-adjusted self-distillation）方法，具体流程如下：首先计算规范调整后的 logit，即将原始 logit 除以对应 token 的嵌入范数，得到 $\text{logit}_{\text{adj},i} = h \cdot e_i / \|e_i\|$，以消除高资源语言的范数偏差；然后对调整后的 logit 应用 top-k/top-p 过滤，得到去偏后的候选 token 集合；接着以多标签方式构建伪标签，若某语言族的任意 token 出现在候选集中，则将该语言族的标签置为 1；最后在冻结 LLM 权重的条件下，以二元交叉熵损失 $\mathcal{L} = \sum_i \text{BCE}(y^*_{t,i},\, \sigma(z_{t,i}))$ 训练 LCG。这种自蒸馏策略的妙处在于，它利用模型自身去偏后的预测作为监督信号，无需任何人工标注的语言标签数据，实现了数据高效的训练。训练数据共约 78,000 条样本，覆盖 200 余种语言，来源包括 Aya Dataset、FLORES+、DeepSeek Distill Dataset 以及中英文 Alpaca 数据。为防止过度干预破坏合理的代码切换，LCG 在推理时设有三条保护规则：符号和低资源 token 永不被屏蔽；若门控预测与模型高置信度输出相矛盾（即即便屏蔽后目标语言 token 概率极低），则不进行干预；同时始终允许前一个非符号 token 所属语言族的 token 通过，以保证语言连贯性。

在实验设计上，论文在非思维链（no-think）和思维链（thinking/reasoning）两类模型上分别进行了系统性评估，涵盖 Qwen3-30B-A3B、Qwen3-8B、Llama3.1-8B、Gemma3-12B、GPT-OSS-20B 等多个主流模型。评估任务包括 FLORES+ 多语言翻译（阿拉伯语、希伯来语、韩语、泰语）、INCLUDE 多语言知识推理基准，以及 Humaneval-XL 代码生成（阿拉伯语和希伯来语题面的 Python 编程题）。为了准确区分错误语言混淆与合法代码切换，论文创新性地将 FLORES+ 的参考译文按是否包含拉丁字符分为 FLORES-NO-LATIN 和 FLORES-WITH-LATIN 两个子集，前者用于度量不应出现的拉丁字母混入，后者用于验证合法代码切换的保留程度。对比基线包括提示工程（ICL）、贪心解码和 ORPO 微调三种方法。

实验结果全面展示了 LCG 的有效性。在非思维链模型上，LCG 对语言混淆的压制效果极为显著：Qwen3-30B-A3B 的中日混淆率从 1.0% 降至 0.0%，拉丁字母混淆率从 4.4% 降至 0.4%，BLEU 分数不降反升（13.2→13.4）；Llama3.1-8B 的中日混淆率从 3.0% 降至 0.4%，拉丁混淆率从 8.4% 降至 2.9%；Qwen3-8B 的中日混淆率从 4.5% 降至 0.1%，拉丁混淆率从 12.1% 降至 2.0%。在 INCLUDE 多语言推理基准上，Qwen3-30B 的中日混淆率从 2.21% 降至 0.11%，而准确率几乎未受影响。在思维链模型的代码生成任务上，Humaneval-XL 结果同样令人满意：Qwen3-8B 的中日混淆率从 1.50% 降至 0.06%，Pass@1 从 67.94% 保持至 67.19%；GPT-OSS 的中日混淆率从 0.38% 降至 0.06%，Pass@1 甚至略有提升（59.88%→60.19%）。对比基线方法，ICL 和贪心解码几乎没有显著改善（混淆率仅从 4.5% 降至 4.2%），ORPO 在某些模型上效果相当但显著损害了通用能力（INCLUDE 准确率从 61.4% 下降至 57.3%），而 LCG 在保持任务性能的同时实现了数量级的混淆率降低。在代码切换保留分析中，LCG 调整后的 Qwen3-8B 在 FLORES-WITH-LATIN 上的代码切换率从 46.34% 降至 25.90%，高于 Claude Sonnet 4 的基线水平（23.29%），并在 token 级别分析中对 86.7% 的自然代码切换位置保持了英文 token 的通过。规范调整消融实验表明，去除范数调整后 LCG 性能一致下降，验证了这一设计的必要性。在计算效率方面，全部实验仅需两块 NVIDIA A100 GPU，训练 Qwen3-30B-A3B 约需 12 小时，轻量级 MLP 的计算开销可与推测解码（speculative decoding）完全并行，几乎不引入额外延迟。

综合来看，LCG 从一个看似简单的工程问题出发，深入剖析了语言混淆的机制根源（嵌入范数偏差），并以优雅的自蒸馏训练框架和精心设计的推理规则，实现了对语言混淆的精准干预。其即插即用的特性使其可以无缝集成到任意 LLM 的推理流程中，无需重新训练底层模型，具有极高的实用价值。论文也坦诚地指出了当前方法的局限性：LCG 目前以书写系统（script）而非具体语言为粒度进行分类，因此无法区分共享同一书写系统的语言之间的混淆（如西班牙语与英语之间），也无法细分不同低资源语言之间的差异。这为未来设计更细粒度的语言感知门控提供了明确的研究方向，也预示着将语言意识引入解码过程这一思路具有广阔的延伸空间。
