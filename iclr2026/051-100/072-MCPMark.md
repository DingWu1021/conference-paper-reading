# MCPMark: A Benchmark for Stress-Testing Realistic and Comprehensive MCP Use

> **Venue**: ICLR 2026 | **ArXiv**: 2509.24002 | **MCPMark**

随着大型语言模型（LLM）能力的持续提升，如何让模型与外部工具和环境进行可靠、复杂的交互已成为智能体研究的核心议题。模型上下文协议（Model Context Protocol，MCP）作为一种标准化接口规范，为 LLM 与外部系统之间的通信奠定了基础，成为构建通用智能体的重要基础设施。然而，尽管 MCP 在工业界和学术界得到了广泛关注，现有的评测基准却在很大程度上停留于较为浅薄的交互层次：它们普遍侧重于"读"操作，任务的平均交互轮数仅为 3.2 至 6.8 轮，既缺乏覆盖创建、更新、删除等完整 CRUD 操作的设计，也难以模拟真实应用场景中多步骤、多服务协作的复杂工作流。这一空缺严重制约了研究者对当前模型在现实世界智能体部署中实际能力与局限性的准确认知。针对上述问题，来自多个研究机构的学者联合提出了 MCPMark 这一全面压力测试基准，并将其收录于 ICLR 2026 会议。

MCPMark 的核心设计哲学是"真实性"与"全面性"的统一。基准共包含 127 个高质量任务，横跨五个主流 MCP 服务环境：Notion（文档与知识管理）、GitHub（代码托管与版本控制）、Filesystem（本地文件系统操作）、PostgreSQL（关系型数据库查询与管理）以及 Playwright（基于浏览器的自动化交互）。这五类环境共涉及 55 个 MCP 服务器，并在 38 个精心设计的初始状态基础上构建任务场景，充分覆盖了从本地服务到远程云端服务的多元化部署形态。每个任务均配备自然语言指令（平均长度达 288.6 词）、程序化验证脚本（平均代码量为 209.8 行）以及明确定义的环境初始状态，形成"指令—执行—验证"的完整评测闭环。

在任务构建方法论上，MCPMark 采用了一套人机协作的多阶段流水线，涉及 10 位领域专家，平均每个任务需要 3 至 5 小时的人工投入。具体流程分为四个阶段：探索阶段，由专家与 AI 智能体协同探索目标环境，发掘具有代表性的操作模式；演化阶段，基于初始任务原型逐步提升任务复杂度和多步性；验证阶段，为每个任务编写并调试程序化验证脚本，确保评测结果的客观可再现性；迭代阶段，在保持可验证性的前提下持续增加难度。全部任务在正式发布前还经过了跨专家交叉审核及为期一个月的社区验证，确保质量与覆盖面的双重保证。相较于依赖 LLM 作为评判者（LLM-as-judge）的主观评估方法，MCPMark 的程序化验证机制从根本上消除了评测误差，为严格的科学比较提供了可靠基础。

在评测框架层面，论文提出了配套的标准化评测工具 MCPMark-Agent，通过工具调用循环驱动模型与 MCP 环境交互，每个任务最多支持 100 轮执行，超时限制为 3600 秒。评测采用三项互补指标：$\text{pass@1}$ 衡量单次运行的成功率，反映模型的平均表现；$\text{pass@4}$ 衡量 4 次独立尝试中至少一次成功的概率，反映模型的最大潜力；$\text{pass}^4$ 要求 4 次运行全部成功，是衡量执行稳定性和一致性的严格指标，对于真实世界的可靠部署尤为关键。这三项指标的组合从不同维度全面刻画了模型在复杂 MCP 使用场景中的综合能力。

实验结果揭示了当前前沿模型面临的严峻挑战。在所有参评模型中，表现最优的 gpt-5-medium 仅达到 52.56% 的 $\text{pass@1}$ 和 33.86% 的 $\text{pass}^4$，而广受认可的强力模型 claude-sonnet-4 和 o3 的 $\text{pass@1}$ 分别仅为 28.15% 和 25.39%，双双跌破 30% 的门槛；它们的 $\text{pass@4}$ 指标更不足 15%。开源模型的最佳成绩由 qwen3-coder-plus 以 24.80% 的 $\text{pass@1}$ 取得。从不同服务环境来看，各模型在 PostgreSQL 上表现相对较好（gpt-5-medium 达 76.19%），而在 Playwright（43.00%）和 Notion（41.96%）等远程服务上的表现明显偏弱，这一规律与训练数据分布及本地环境模拟保真度密切相关。模型在 MCPMark 上平均需要 16.2 轮执行和 17.4 次工具调用才能完成一个任务，相比现有基准的 3.2 至 6.8 轮，压力测试强度显著提升。

推理力度对模型性能的影响同样是本文重要发现之一。以 gpt-5 为例，将推理力度从低档提升至中档，$\text{pass@1}$ 从 46.85% 升至 52.56%；而对于 gpt-5-mini，从低档提升至高档则带来了从 8.27% 到 30.32% 的大幅跃升，涨幅超过 22 个百分点。这一规律在远程服务上尤为显著，而本地服务（如 Filesystem、PostgreSQL）对推理力度提升的敏感性相对较低，暗示远程服务任务对深度推理和上下文管理提出了更高要求。从稳定性角度看，$\text{pass@1}$ 与 $\text{pass}^4$ 之间存在显著落差，说明即便是最优模型也面临严重的一致性问题——同样的任务有时成功有时失败，这对实际部署中的可靠性构成严峻考验。

在失败模式分析方面，论文将失败分为两大类：隐性失败（implicit failures）指任务表面完成但验证未能通过，成因包括推理错误、规划不当或上下文处理失当，占全部失败的 50% 以上；显性失败（explicit failures）则包括上下文窗口溢出、达到轮次上限、提前放弃任务以及工具调用格式错误等。值得注意的是，不同模型的失败模式存在差异：gpt-5-high 的失败中有超过 80% 属于隐性失败，而 gemini-2.5-flash 的显性失败比例则高达 48%，反映了不同模型架构在错误处理策略和执行稳定性上的本质差异。成功执行与失败执行之间的另一个显著区别在于工具调用效率：成功的完成往往伴随着精准、有目的性的工具调用序列，而失败则常见重复性或探索性的冗余调用，说明当前模型在长程规划和状态追踪方面仍有较大提升空间。

MCPMark 的研究意义在于从多个维度重新界定了 MCP 智能体评测的标准。首先，它以完整 CRUD 操作覆盖和远超现有基准的任务复杂度，揭示了当前模型在真实工作流中的能力天花板；其次，程序化验证机制为公平、可重现的模型比较提供了客观保证；再者，三维评测指标体系让研究者能够同时审视模型的平均性能、潜在上限与稳定可靠性。从更宏观的视角来看，论文指出当前前沿模型若要真正胜任现实世界中复杂 MCP 智能体的部署需求，至少需要在三个方向取得突破：一是从被动响应式的工具调用演进至具备深度前瞻推理能力的主动规划；二是提升上下文利用效率与历史信息管理能力，有效应对长对话场景下的记忆衰减；三是构建稳健的错误处理与恢复机制，保障多步骤任务的端到端执行一致性。MCPMark 作为一个开放、可扩展的评测平台，为这一研究方向提供了坚实的基础设施，其高质量的任务集、标准化的评测工具链以及来自真实 MCP 生态的多样环境覆盖，必将持续推动智能体能力评测与模型研发的协同进步。
